# 业务闭环

## 1. 端到端流程

1. 用户提交目标，API 完成身份、角色、请求大小和速率校验。
2. Input Guard 检测越权、注入、凭据访问和破坏性意图。
3. Agent 分类意图和复杂度，确定性规则复核风险下限。
4. 系统只在允许范围采集上下文并检索已审核知识。
5. LLM 生成严格 Schema 的候选 `ActionPlan`；解析失败不得执行。
6. Plan Guard 校验目标一致性、模式、Tool、参数、路径、角色和资源范围。
7. MCP 只读 Tool 采集证据；所有返回按 `UNTRUSTED_DATA` 封装。
8. RCA 使用规则权重、异常程度、时间关联、证据数量和案例相似度生成 Top 3。
9. 只读任务返回证据、结论、不确定性和建议。
10. 写任务执行 Action Guard、dry-run 和影响摘要。
11. L3 进入人工审批；L4 永久拒绝。
12. 审批令牌绑定用户、任务、Tool、参数哈希、nonce 和有效期。
13. Restricted Executor 执行备份、固定 argv 动作和独立验证。
14. 验证失败自动回滚并再次验证。
15. 所有阶段写入脱敏、可校验的审计事件。

## 2. 不变量

- LLM 输出不能直接进入 Executor。
- 不存在任意 Shell Tool 或命令字符串执行接口。
- READ_ONLY 永远不产生系统状态变更。
- L3 没有有效审批令牌就不能备份或执行。
- L4 不能通过人工审批转为可执行。
- 备份失败即停止；验证失败必须尝试既定回滚。
- 日志或知识中的指令不能改变系统策略。
- 只记录结构化决策链，不记录隐藏思维。

## 3. RCA 输出

每个根因候选包含 `title`、确定性计算的 `confidence`、`evidence_ids`、公开 `reason_summary` 和 `recommended_actions`。证据不足时必须明确返回“不足以确定”，不得由模型补造证据或置信度。

## 4. 降级闭环

- LLM 不可用：确定性只读能力继续；需要规划的任务明确阻塞。
- MCP 不可用：任务进入 `BLOCKED`，不生成模拟真实证据。
- 演示离线夹具：必须标记 `offline_fixture=true`，规则、审批、Tool 和审计仍走真实闭环。
- 验证或回滚失败：停止任务并生成事件，不扩大执行范围。
