# ADR-0001：初始技术选型

- 状态：Accepted for planning
- 日期：2026-07-11
- 范围：Phase 0 架构决策，不代表实现或真机验证

## 决策

1. 后端采用 Python 3.10+、FastAPI、Pydantic、SQLAlchemy、Alembic、httpx、PyYAML、pytest、ruff 和 mypy。
2. 数据库基线采用 SQLite WAL 与 FTS5；向量数据库不是硬依赖。
3. 前端采用 Vue 3、TypeScript、Vite、Pinia、Vue Router、Element Plus 和 ECharts。
4. 系统能力通过官方 MCP Python SDK 和固定 Tool 暴露；内部 Registry 与 MCP 传输适配分离。
5. 系统命令只允许固定 argv 与 `shell=False`；不存在通用 Shell Tool。
6. DeepSeek 是默认 LLM Provider，但模型、Base URL 和能力配置化；Provider 接口预留其他模型。
7. 确定性 Guardrail 独立于模型，拥有最终否决权；Restricted Executor 不接受模型原始输出。
8. 默认运行模式为 READ_ONLY；L3 只在 CONTROLLED_EXECUTION 且人工审批后执行。
9. Docker 仅可选，不作为麒麟 LoongArch 唯一部署方式。
10. 后续页面参考 `claude-code-system-prompts/` 的信息设计原则，但不引入 Claude 专属协议、品牌或隐藏思维展示。

## 理由

该组合优先满足安全确定性、可复现闭环、轻量部署、LoongArch 兼容和 7 分钟演示稳定性。SQLite/FTS5 降低部署复杂度，Provider 与 MCP 适配层降低厂商和协议变化风险。

## 后果

- Phase 1 需建立严格配置、数据库和状态机基础。
- Phase 2/6 必须验证每个 Tool 的固定 argv、权限和回滚契约。
- Phase 9 必须在麒麟 V11 LoongArch 真机验证依赖、系统命令和性能。
- 当前 Windows 环境只能标记 `DESIGNED`，不得声明 `KYLIN_VERIFIED` 或 `LOONGARCH_VERIFIED`。
